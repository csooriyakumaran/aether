  ## Roadmap toward v0.0.3 and beyond

### Features (new capability)

  - [x] `arena_read_file` helper ‚Äî the README's own "Temporary File Processing" use case hand-rolls fopen/fseek/ftell/fread into an arena; that's common enough to be a real function instead of copy-pasted boilerplate each time.
  - [.] str8 operations ‚Äî equality/comparison, slicing, trim done (`str8_eq`/`str8_cmp`/`str8_slice`/`str8_trim`, tested in `tests/test_strings.c`); find/split still open. find: out-param + `b8` return (matches `os_file_size`'s convention) vs. an `npos`-style sentinel. split: non-allocating iterator (`str8_split_next` over a cursor) vs. an allocating `arena_push_str8_split` returning an array of views ‚Äî leaning iterator-first since it composes better with a parsing loop and the allocating version can always be built on top later.
  - [ ] POSIX backend for the OS layer ‚Äî `os_mem_reserve/commit/decommit/release/pagesize` are Windows-only by design for now. The arena logic itself doesn't change, just those five shim functions, whenever CFD/HPC work lands you on  Linux.
  - [x] add `os_file_map` / `os_file_unmap` using `CreateFileMapping`/`MapViewOfFile` for direct mem-mapping of OS files. (for WIN32 for now)
  - [x] ring / circular buffers ‚Äî virtually-mirrored ("magic") ring buffer on Win10 1803+. `VirtualAlloc2`/`MapViewOfFile3`/`UnmapViewOfFile2` are resolved at runtime via `GetProcAddress` from `kernelbase.dll`, so there is no extra link dependency and the binary still loads on older Windows (alloc just returns `false`). Tested across mirror/seam/stream/counter-wrap in `tests/test_ring.c`.
  - [ ] add `os_file_open_for_write` and `os_file_open_ex` (for fine control)

### Quality of life (easier to use, same capability)
  - [x] Scratch/temp-arena helper (`ArenaTemp` + `arena_temp_begin`/`arena_temp_end`) ‚Äî wraps the manual `u64 mark = arena.pos; ... arena_pop_to(&arena, mark);` pattern you already do by hand everywhere.
  - [x] `CHANGELOG.md` ‚Äî now that real tags exist, worth more than `gh release --generate-notes` alone once there's more than one release to compare across.
  - [x] Release automation (`scripts/release.ps1`) ‚Äî mechanical Phase-B driver: pre-flight guards (on `master`, clean tree, tag absent locally+on `origin`, `gh` authenticated, target `## [X.Y.Z]` changelog section present), `ctest` gate, then drop `-dev` ‚Üí commit `release: vX.Y.Z` ‚Üí annotated tag ‚Üí push ‚Üí `cmake --install` + `Compress-Archive` ‚Üí `gh release create` with notes sliced from the curated `CHANGELOG.md` section ‚Üí re-bump to the next `-dev`. `-DryRun` runs the read-only pre-flight for real but narrates every mutating step. Content edits (CHANGELOG/README/TODO) stay **manual by design** ‚Äî the script *verifies* the changelog section exists rather than generating prose.
  - [ ] GitHub Actions CI ‚Äî build + `ctest` on push/PR. Cheap now that there's a real test suite; catches regressions before they reach a tag. Distinct from the release script above: that gates *releases*, this gates *every push/PR*. Could later share the build+test step (the release script's gate is the same `cmake`/`ctest` invocation a CI job would run).

  ### Hardening / process
  - [ ] Decide an error-handling convention before adding more I/O-shaped APIs. Arenas now hard-abort on OOM ‚Äî a fine, deliberate choice for memory. `arena_read_file` and friends will hit a *different* failure shape (missing file, permission denied) that isn't obviously "abort-worthy" the same way. Decide the convention once, not per-function.
  - [ ] Verify actual C++ compilation. The README's first line promises "c/c++ projects," and `extern "C"`/`ARENA_ALIGN` already account for it, but `CMakeLists.txt` only declares `LANGUAGES C` ‚Äî nothing has actually compiled this as C++ yet to confirm the claim holds.
  - [x] Test coverage for the str8 macros (`STR`/`STR8_ARG`/`STR8_FMT`) and string manipulation (`str8_eq`/`cmp`/`slice`/`trim`, `c_str`, `view_from_str8`/`view_from_bytes`) ‚Äî covered in `tests/test_strings.c`. (`cstr8` no longer exists; replaced by `str8_view` in v0.0.5.)
  - [ ] Test coverage for `BIT()`/`BIT8/16/32/64()` ‚Äî the one remaining untested macro group; everything else now has coverage across `tests/test_arenas.c` and `tests/test_strings.c`.
  - [ ] Keep an eye on `ArenaFlags` width (`u8`, room for 4 more flags) ‚Äî already a known todo in the header; don't let flag #5 become an afterthought.
  - [x] ~~add guards against windows version for ring buffers (`_WIN32` && `NTDDI_VERSION >= NTDDI_WIN10_RS4`)~~ ‚Äî handled at *runtime* instead: the `GetProcAddress` probe returns `false` on pre-1803 Windows (more precise than a version number, and degrades gracefully). The `MEM_*_PLACEHOLDER` constants are `#ifndef`-supplied so the header still compiles against an older SDK. A hard compile-time `NTDDI` opt-out (to drop the feature entirely on old *targets*) remains optional and unneeded for now.
  - [ ] POSIX backend for ring buffers ‚Äî same two-views-of-one-mapping trick via `memfd_create` + double `mmap` (or `shm_open`/`ftruncate`), to land alongside the arena's pending POSIX shims.
  - [ ] `ring_buffer_read`/`write` should guard `NULL rb` for parity with `ring_buffer_peek` (they currently check only `rb->base`).
  - [ ] consider an opt-in overwrite-oldest write mode ‚Äî `ring_buffer_write` currently rejects when full; a lossy/overwrite policy is handy for telemetry-style streams that prefer freshest data over backpressure.
  - [ ] **make the ring buffer safe for cross-thread SPSC use** ‚Äî the intended use (producer thread fills, consumer thread drains to a socket) touches `read`/`write` from two threads, but they are plain `u64` today, which is a data race / UB. Promote the cursors to atomics with release/acquire ordering (producer publishes `write` with release *after* the data copy; consumer loads it with acquire *before* reading, then publishes `read` with release *after* consuming; producer loads `read` with acquire *before* overwriting), or guard the cursor updates with a mutex. Required before the zero-copy `peek` ‚Üí `send` ‚Üí `ring_buffer_advance_read` path is used across threads ‚Äî that path's safety already relies on the producer never passing `read`.
  - [ ] (optional) decouple the write side from the read cursor to allow multiple independent readers ‚Äî move cursor ownership out of the buffer so each reader keeps its own `read`, turning the SPSC FIFO into an SPMC broadcast ring. Consequence: the buffer no longer knows what is "unread", so the write path must change from *reject-when-full* to *overwrite-oldest* (the opt-in overwrite item above becomes mandatory here), and each reader detects it was lapped by comparing its cursor against `write - size`. Only worth it if the two-sink (UDP/TCP) fan-out needs a single shared source rather than one ring per sink ‚Äî see that tradeoff before committing to it.
